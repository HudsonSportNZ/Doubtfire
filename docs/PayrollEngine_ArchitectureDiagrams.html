01
High-Level System Architecture
Frontend portals, API layer, domain services, data layer, external integrations
Architecturev1 Scope
Loading diagram...
Architecture pattern: Modular monolith on AWS ECS Fargate. All three portals share a single Fastify API - access is determined by the resolved scope context on each request. Async work (calculation, payslips, filing) runs through BullMQ queues backed by ElastiCache. PostgreSQL with RLS enforces strict tenant isolation at the database level. v1 ships only the Platform Admin Portal; all backend endpoints support all three portals from day one.
02
Domain Entity Hierarchy
Platform to Bureau to Tenant to Employee and supporting domain entities
Domain Model
Loading diagram...
Key isolation rule: Employees belong to Tenants, not Bureaus. Bureau is a management and access layer only. All payroll records are always tenant-scoped. Leave Profiles group leave rules by cohort - employees reference a profile, not individual rules, so policy changes propagate across all affected employees simultaneously.
03
Identity and Access Model
Users, memberships, portal links, scope resolution
Access ControlSecurity Critical
Loading diagram...
Critical distinction: A User and an Employee are not the same entity. An employee only gets portal access when a tenant admin explicitly creates an employee_portal_link - never automatic. A single user can hold memberships at multiple levels simultaneously. Platform-scope users bypass RLS; all other scopes are enforced independently by both application logic and Postgres RLS policies.
04
Portal Architecture
Three portals, one backend, scope-determined access
FrontendAPI-First
Loading diagram...
One API, three experiences: All portals call the same REST endpoints. The scope resolver determines what data each request can access. A platform_admin and a client_owner calling the same endpoint receive different scoped results. Activating the Client and Employee portals in Phases 2 and 3 is a frontend routing change - not a new backend system.
05
Rules Engine Architecture
Five-level inheritance, version control, evaluation context
Rules EngineCore Logic
Loading diagram...
Client-invisible rules: Clients never interact with the rules engine directly. Their payroll settings and leave policies in the Client Portal map to Level 4 rule overrides behind the scenes. Level 1 rules (PAYE, KiwiSaver minimums) are locked. Every pay run stores the exact rule_version_ids used, making all historical payslips exactly reproducible.
06
Payroll Calculation Flow
End-to-end sequence from pay run creation to payslip delivery
SequenceAsync and Idempotent
Loading diagram...
Idempotency guarantee: All writes use X-Idempotency-Key. The BullMQ job_id is derived from the pay_run_id - re-queuing is a no-op. If calculation fails for individual employees, their item is marked status=error and the run continues. Once finalised, the pay run is permanently locked. Corrections are new adjustment pay runs. Calculation snapshots are immutable from creation.
07
Payroll Data Model Overview
Key table relationships across the payroll domain
Data ModelSchema Reference
Loading diagram...
Three-layer payroll output: Each calculated employee produces a calculation_snapshot (immutable audit record), a pay_run_item (bank payment reconciliation summary), and multiple pay_run_line_items (one per component, used for payslips and reports). Leave balances are never stored - they are computed from the append-only leave_transactions ledger.
08
Reporting Architecture
pay_run_line_items as the universal reporting layer
ReportingMulti-level
Loading diagram...
Why pay_run_line_items: All reporting queries run against the structured pay_run_line_items table rather than parsing raw JSONB snapshots. Each row is a single component with a typed code field (e.g. PAYE, KIWISAVER_EE, ORDINARY_HOURS). Every report type becomes a simple, fast, well-indexed SQL query.
09
Request Scope Resolution
From login to permission enforcement, step by step
Auth FlowSecurity Critical
Loading diagram...
Defence in depth: Scope resolution happens at three independent layers - JWT validation (cryptographic), scope context from membership tables (application logic), and Postgres RLS (database-level). A bug in application logic cannot leak cross-tenant data if RLS is correctly configured. The scope context is passed to every service method and query, ensuring no code path can accidentally access out-of-scope records.
